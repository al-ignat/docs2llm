# docs2llm

## Git Workflow

- Always create a feature branch before committing. Don't commit directly to main — set up for PRs.

## Commands

```bash
bun run src/commands/cli.ts   # Run CLI in dev mode
bun test                       # Core tests (167 tests, uses test/ dir via bunfig.toml)
bun run typecheck              # TypeScript type checking (tsc --noEmit)
bun run build                  # Compile standalone binary → ./docs2llm
bun run build:raycast          # Build Raycast extension (sandboxed)
cd raycast && npm run build    # Build Raycast extension (direct)
cd raycast && npm test         # Raycast tests (193 vitest tests)
bun run verify                 # Run everything: typecheck + tests + raycast tests + both builds
```

## Stack

- Bun + TypeScript, no build step. Run with `bun run cli.ts`.
- Kreuzberg for inbound extraction, Pandoc for outbound conversion.
- Interactive mode uses @clack/prompts.

## Architecture

### `src/commands/` — CLI entry points and wizards
- `cli.ts` — arg parsing, help text, entry point, command routing
- `run.ts` — non-interactive conversion orchestration (single file, folder, URL, stdin)
- `interactive.ts` — interactive mode (autocomplete file picker, format picker, batch conversion)
- `init.ts` — `docs2llm init` wizard (defaults, templates, config diff preview)
- `config-wizard.ts` — `docs2llm config` wizard (view, edit, create configs)
- `paste.ts` — `docs2llm paste` (clipboard → Markdown)
- `fetch.ts` — URL fetching for `docs2llm <url>`
- `watch.ts` — `docs2llm watch` file watcher

### `src/core/` — business logic (no UI)
- `convert.ts` — dispatches to inbound (Kreuzberg) or outbound (Pandoc)
- `outbound.ts` — Pandoc wrapper
- `validate.ts` — validation layer (direction detection, smart defaults, collision checks)
- `config.ts` — YAML config loading, merging, template resolution
- `output.ts` — output path resolution and file writing
- `tokens.ts` — token counting, LLM fit checks, chunking
- `scan.ts` — file discovery for interactive mode
- `clipboard.ts` — clipboard read/write (pbcopy/pbpaste, xclip)
- `url-safe.ts` — URL sanitization and safe fetching

### `src/server/` — web UI and MCP
- `api.ts` — HTTP API routes
- `ui.ts` — web UI HTML/CSS/JS
- `mcp.ts` — MCP server for Claude Desktop / Cursor

### `src/shared/` — cross-cutting utilities
- `wizard-utils.ts` — `guard()` cancel helper for @clack/prompts
- `wizard-steps.ts` — step tracker for multi-step wizards

### `raycast/src/` — Raycast extension
- `convert-file.tsx`, `convert-clipboard.tsx`, `quick-convert.tsx` — view commands (Form → result)
- `smart-copy.tsx`, `smart-paste.tsx`, `smart-save.tsx` — no-view commands (auto-detect → action)
- `lib/docs2llm.ts` — CLI integration (resolveInvocation, convertFile, exportMarkdown, etc.)
- `lib/format-utils.ts` — shared direction detection, format constants, format titles
- `lib/result-view.tsx` — shared result display component
- `lib/clipboard-detect.ts` — clipboard content type detection
- `lib/smart-detect.ts` — source cascade (Finder → selection → clipboard)
- `__tests__/*.test.ts` — 7 vitest test files (smart-*, format-utils, convert-clipboard-logic, package-structure)

## Raycast Extension

- Build: `cd raycast && npm run build`
- Test: `cd raycast && npm test` (vitest)
- `package.json` defines commands + per-command preferences — update it first when adding/removing commands
- `raycast-env.d.ts` is auto-generated by `ray build` — don't edit manually
- When adding params to lib functions, update test assertions in `__tests__/smart-*.test.ts`

## Compound Engineering Workflow

This project uses the CE framework for structured feature development:

1. **Brainstorm** (`/ce:brainstorm`) — Explore what to build. Output: `docs/brainstorms/YYYY-MM-DD-<topic>-brainstorm.md`
2. **Plan** (`/ce:plan`) — Design how to build it. Output: `docs/brainstorms/YYYY-MM-DD-<topic>-plan.md`
3. **Work** (`/ce:work`) — Execute the plan. Branch → implement → PR.
4. **Review** (`/ce:review`) — Multi-agent code review.
5. **Compound** (`/ce:compound`) — Document learnings. Output: `docs/solutions/YYYY-MM-DD-<topic>.md`

### docs/ structure
- `docs/brainstorms/` — Active brainstorms and plans (living docs)
- `docs/solutions/` — Solved problems and learnings (institutional knowledge)
- `docs/archive/` — Completed docs from past work (security audits, research, old plans)

## Gotchas

- macOS filesystem is case-insensitive. `ROADMAP.md` and `roadmap.md` are the same file. Always check existing files before creating new ones.
- `bunfig.toml` sets `[test] root = "./test"` — `bun test` only runs core tests. Raycast tests use vitest and must be run separately with `cd raycast && npm test`.
